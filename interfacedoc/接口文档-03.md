## 时间槽管理

前缀都为 `/creator`：

| 业务动作       | Controller 方法 | 路径          | 方法 | 入参                     | 出参类          | 状态 |
| -------------- | --------------- | ------------- | ---- | ------------------------ | --------------- | ---- |
| 批量生成时间槽 | `saveSlotBatch` | `/slots`      | POST | `List<SlotDTO>`          | `Void`          |      |
| 分页查询时间槽 | `getSlotPage`   | `/slots/page` | GET  | `SlotPageQuery`          | `IPage<SlotVO>` |      |
| 修改时间槽状态 | `updateSlot`    | `/slots/{id}` | PUT  | `路径参数 id` + `status` | `Void`          |      |

### 批量生成时间槽

```json
// SlotDTO：单个时间槽
{
  "startTime": "2026-04-01 10:00:00",     // 开始时间（必填）
  "endTime": "2026-04-01 11:00:00",       // 结束时间（必填）
  "price": 50,                            // 预约价格（虚拟币，必填）
  "address": "腾讯会议链接xxx"             // 预约地址（必填）
}
```

```mermaid
sequenceDiagram
    participant Client as 客户端
    participant Auth as 认证服务
    participant DB as 数据库

    Client->>Auth: POST /creator/slots<br/>List<SlotDTO>
    Auth->>Auth: 从 Spring Security 上下文获取 userId
    Auth->>Auth: 校验用户是否为创作者
    alt 非创作者
        Auth-->>Client: 403 Forbidden
    else 是创作者
        loop 遍历每个 SlotDTO
            Auth->>DB: 检查时间槽是否与已有槽位冲突
            alt 冲突
                Auth-->>Client: 409 Conflict<br/>{msg: "时间槽与已有槽位冲突"}
            end
        end
        Auth->>DB: 批量插入时间槽记录（状态：AVAILABLE）
        Auth-->>Client: 200 OK
    end
```

### 修改时间槽状态

  `status` 有 `CANCELLED` / `AVAILABLE`（关闭或开启）

```mermaid
sequenceDiagram
    participant Client as 客户端
    participant Auth as 认证服务
    participant DB as 数据库

    Client->>Auth: PUT /creator/slots/1?status=CANCELLED
    Auth->>Auth: 从 Spring Security 上下文获取 userId
    Auth->>Auth: 校验用户是否为创作者
    Auth->>DB: 校验时间槽是否存在且属于当前用户
    alt 不存在或无权限
        Auth-->>Client: 404 Not Found
    else 存在
        alt 状态已是 BOOKED
            Auth-->>Client: 400 Bad Request<br/>{msg: "已被预约，无法修改"}
        else
            Auth->>DB: 更新时间槽状态
            Auth-->>Client: 200 OK
        end
    end
```

## 订单状态图

前排提醒，后面将涉及用户、创作者和管理员的操作，还要考虑退款情况，订单状态将很复杂，需要先看看下面状态图：

> 为了简化，这里不支持第三方支付，而是虚拟币直接扣除。

```mermaid
stateDiagram-v2
    [*] --> PENDING_CONFIRM: 用户发起预约（扣款成功）
    
    PENDING_CONFIRM --> CONFIRMED: 创作者确认
    PENDING_CONFIRM --> CANCELLED: 创作者拒绝/用户取消/超时未确认
    
    CONFIRMED --> COMPLETED: 创作者标记完成
    CONFIRMED --> REFUND_PENDING: 用户申请退款
    
    COMPLETED --> REFUND_PENDING: 用户申请退款（质量问题）
    
    REFUND_PENDING --> REFUNDED: 管理员通过
    REFUND_PENDING --> CONFIRMED: 管理员拒绝（CONFIRMED 来源）
    REFUND_PENDING --> COMPLETED: 管理员拒绝（COMPLETED 来源）
    
    COMPLETED --> [*]
    CANCELLED --> [*]
    REFUNDED --> [*]
```

退款情况：

| 场景         | 订单状态          | 触发条件                       | 退款方式                               |
| ------------ | ----------------- | ------------------------------ | -------------------------------------- |
| 用户主动取消 | `PENDING_CONFIRM` | 用户点击取消                   | 全额自动退款                           |
| 创作者拒绝   | `PENDING_CONFIRM` | 创作者点击拒绝                 | 全额自动退款                           |
| 超时未确认   | `PENDING_CONFIRM` | 创作者48小时内未确认           | 全额自动退款                           |
| 超时未完成   | `CONFIRMED`       | 预约时间已过但创作者未标记完成 | 用户可申请退款，或系统自动发起退款申请 |
| 用户申请退款 | `CONFIRMED`       | 用户发起申请                   | 管理员审核                             |
| 咨询质量问题 | `COMPLETED`       | 特殊情况                       | 管理员审核                             |

也不用一直惦记，后面时序图会说明什么时候退款的，不过要能看懂代码。

### 确认预约

```mermaid
sequenceDiagram
    participant Client as 客户端
    participant Auth as 认证服务
    participant DB as 数据库

    Client->>Auth: PUT /creator/consult/{id}/confirm<br/>{address: "新会议链接（可选）"}
```
